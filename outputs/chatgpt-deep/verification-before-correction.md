# chatgpt-deep 修正前検証

## 検証対象

- `first-output.md`
- 静的確認のみ。Excel実行はIssue #6で行う。

## 良い点

- `ThisWorkbook.Path\daily\YYMM\` 配下の再帰探索になっている。
- 月次ブック名、日次ファイル名、日付妥当性を分けて判定している。
- 同一日付の重複ファイルはフルパス昇順で1件だけ処理し、残りをログする設計になっている。
- ログ出力では `GetRelativePath` を使い、絶対パスを残しにくい設計になっている。
- `Scripting.Dictionary` と `FileSystemObject` をlate bindingで使っており、追加参照設定を不要としている。
- `Application` 状態を退避・復元している。

## 主な懸念

1. 0件の扱いが期待結果と合わない可能性が高い。
   - `RecalculateMonthlyTotalsAndSubtotals` は、0件でも `DETAIL` の月合計や `SUBTOTAL` の日別/合計へ `0` を書く。
   - サンプル期待結果では、0件のセルは空欄として扱われている。
   - 特に全件0のDETAIL行や、発生がない日付の小計列で差が出る。

2. 初期クリア対象が仕様より広い。
   - 仕様では、転記前に `DETAIL` 行の `H:AL` をクリアする。
   - 初回出力は `DETAIL` と `SUBTOTAL` の `H:AL` と `AM` をクリアする。
   - 最終的に再計算しているが、仕様上の人手介入境界としては広すぎる。

3. マスタ読込の範囲が混ざる。
   - `BuildMasterCodeSets` は、見つけた `支店コード` 列または `業務コード` 列について、該当列の最終使用行まで読み続ける。
   - サンプルの `マスタ` は、支店マスタ、業務マスタ、支店・業務対応表が縦に並ぶため、この方式では支店コード集合に業務コードや見出しが、業務コード集合に支店コードや見出しが混ざる。
   - 既知支店/業務判定が甘くなり、異常分類の精度が落ちる可能性がある。

4. 対象拡張子の扱いが広い。
   - `EnumerateDailyFolder` は `.xls` で始まる拡張子を拾い、後段で `クレーム集計YYMMDD.xlsx` 以外をinvalidとしてログする。
   - 仕様の対象は `.xlsx` なので、`.xlsm`, `.xlsb`, `.xls` は最初から対象外にする方がログが明確になる。

## 修正依頼方針

1回だけの修正依頼では、最終結果の一致に直結する0/空欄、小計、クリア範囲、マスタ範囲、対象拡張子をまとめて指摘する。
